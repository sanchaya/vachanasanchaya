Rails 3.2.22
Sqlite

To start solr
bundle exec rake sunspot:solr:start

To reindex Solr
rake sunspot:solr:stop
rm -rvf solr (Delete your /solr directory)
rake sunspot:solr:start
rake sunspot:solr:reindex

=== CPU performance review (June 2026) ===

==== Critical fixes ====

1. HomeController#index random vachana selection +
   Before: +Vachana.pluck(:id).sample+ loaded ALL vachana IDs into memory (~25k),
   then looped +Vachana.find(rand)+ until a non-blank vachana was found.
   After: Single SQL query with +WHERE vachana != '' ORDER BY RAND() LIMIT 1+.

2. AiSearchService — batched keyword queries +
   Before: Called +KeyWord.search_vachana_pada()+ once per search word (N queries).
   After: Single +KeyWord.where(word: words)+ for exact matches, single +LIKE+ for
   partial matches.

3. VachanasController#index — deferred WordList/Vachanakaara loading +
   Before: +WordList.all+ and +Vachanakaara.all+ loaded on every index request even
   when no search was performed.
   After: Moved inside the search +if+ branch.

4. Vachana.search_vachana_pada — eliminated N+1 COUNT queries +
   Before: Per-vachanakaara COUNT queries + individual +vachana.vachanakaara+ loads
   (3 queries per author in results).
   After: Single +GROUP BY+ for word counts, batched +Vachanakaara.where(id: ...)+
   loaded once.

5. Letter counting moved from Ruby to SQL +
   Before: +pluck(:vachana).compact.each+ loaded all vachana texts into memory for
   letter counting in +vachanas_controller+ and +vachanakaaras_controller+.
   After: +group("LEFT(vachana, 1)").count+ — counting done entirely in SQL.

6. Added database indexes (migration 20260616000000) +
   - +glossaries.word+ (used in +start_letter+, +words_like+ scopes)
   - +word_lists.name+ (used in +find_by_name+)
   - +roles.name+ (used in authorization queries)
   - +review_comments.review_vachana_id+ and +review_comments.user_id+
   - +daily_vachanas.created_at+ (used in date comparison)

7. Enabled Rack::Attack rate limiting +
   Before: All throttling rules were commented out — no protection against abusive
   clients or scrapers consuming CPU.
   After: Enabled 300 req/5min per-IP throttle and login rate limiting.

8. Sunspot/Solr search pagination +
   Before: +@search.results+ returned every matching record without limit.
   After: Added +paginate :page => ..., :per_page => 50+.

9. Vachana.vachanakaara_ids — avoid loading entire association +
   Before: +includes(:vachanakaara).map(&:vachanakaara_id)+ loaded all associated
   records just to extract IDs.
   After: +pluck(:vachanakaara_id).uniq+.

10. KeyWord.vachanakaara_keyword — cached Vachanakaara lookups +
    Before: Called +Vachanakaara.find_by_id+ individually for every keyword's
    vachanakaara_ids (N+1).
    After: In-memory hash cache avoids repeated lookups.

=== Serialized columns migrated to join tables (June 2026) ===

Summary: Replaced 3 YAML-serialized TEXT columns with proper normalized join tables,
eliminating deserialization overhead on every read and enabling indexed SQL queries.

+--------------+---------------------+---------------------+------------+
| Old column   | Format              | New table           | Rows moved |
+--------------+---------------------+---------------------+------------+
| vachana_ids  | Hash {id => count}  | keyword_vachanas    |    614,293 |
| vachanakaara | Array [id, ...]     | keyword_vachanakaar |    373,544 |
| ids (concord | Array [id, ...]     | concord_items       |    146,253 |
+--------------+---------------------+---------------------+------------+

Files changed: 18 files, +219 / -84 lines. Backups: full MySQL dump (14MB gzip),
raw CSV exports, separate git branch.

Changes made:
- +db/migrate/20260616000001_create_join_tables.rb+ — created three join tables with
  unique composite indexes and foreign key indexes.
- +app/models/keyword_vachana.rb+, +keyword_vachanakaara.rb+, +concord_item.rb+ —
  new ActiveRecord models.
- +app/models/key_word.rb+ — added +has_many+ associations (+keyword_vachanas+,
  +keyword_vachanakaaras+, +linked_vachanas+, +linked_vachanakaaras+). Added helper
  methods (+all_vachana_ids+, +vachanakaara_id_list+, +vachana_id_count_hash+,
  +vachana_count_for+). Refactored +self.vachanas+ class method and
  +self.vachanakaara_keyword+ to use join tables.
- +app/models/concord.rb+ — added +has_many :concord_items+ and +item_id_list+ helper.
- +app/controllers/researches_controller.rb+ — uses +all_vachana_ids+ and
  +keyword_vachanas.count+ instead of +vachana_ids.keys+ and +vachana_ids.count+.
- +app/controllers/vachanas_controller.rb+ — uses +vachana_id_count_hash+ instead of
  +vachana_ids.each+.
- +app/services/ai_search_service.rb+ — uses +all_vachana_ids+ instead of +vachana_ids.keys+.
- +app/views/researches/show.html.erb+ and +_results.html.erb+ — use association counts
  instead of serialized column methods.
- +lib/tasks/keywords.rake+ — all three concordance tasks and the keyword count task now
  write to join tables directly. The +update_keyword_count_in_vachana_priority_1+ task
  uses +keyword_vachanas.create!+ and +keyword_vachanakaaras.create!+ instead of
  manipulating serialized hashes/arrays.
- +lib/tasks/import.rake+ — added +migrate_keyword_join_tables+, +migrate_concord_join_table+,
  and +migrate_all_serialized_data+ rake tasks for future data migrations.

Benefits:
- No YAML deserialization overhead on every read (was parsing TEXT → Hash/Array in Ruby)
- Proper indexing on foreign keys — fast JOINs instead of full-table scans
- Type safety — integers stored as integers, not YAML strings
- Queryable via SQL — can find all keywords for a vachana without scanning every row
- Eliminates duplicate entries (vachanakaara_ids had duplicates from naive << append)
- The +concord.ids+ column was never read by application code — now properly normalized

==== Remaining architectural recommendations ====

- Install Sidekiq/Delayed Job for CSV/ZIP downloads that currently load all
  records synchronously (+Vachana.download_all+, +Vachana.all_vachanas+).
- Add APM monitoring (New Relic, Scout, or AppSignal) to detect regressions.
- Upgrade from Rails 3.2.22 (EOL June 2016) for Active Job, better caching,
  and security patches.
- Drop old serialized columns (+vachana_ids+, +vachanakaara_ids+, +ids+) after
  verifying the new join tables work correctly in production.

=== Text-to-Speech (TTS) — listen to vachanas (June 2026) ===

Summary: Added Kannada text-to-speech playback using the Web Speech API
(+SpeechSynthesisUtterance+) — zero dependencies, zero cost.

Added to:
- Vachana show page (+app/views/vachanas/show.html.erb+)
- Search result modals (+app/views/vachanas/_vachana_concord_results.html.erb+)
- Concord result modals (+app/views/vachanas/_results.html.erb+)

JavaScript in +app/assets/javascripts/vachanas.js+:
- +speakVachana(elementId)+ — reads text from a DOM element and speaks it.
- +speakVachanaText(text)+ — speaks arbitrary text.
- Voice selection: searches for a Kannada voice (+kn+) first, falls back to
  any Indian English voice (+IN+), then any available voice.
- Handles the +voiceschanged+ race condition (voices not loaded at page load
  in Chrome — waits for the event before first utterance).
- Rate: 0.85 (slightly slower than default for better clarity).

Browser support: Chrome, Edge, Safari. Firefox lacks Kannada TTS voices.

Note: Users must have a Kannada TTS voice installed on their system.
- Windows: Settings → Speech → Add voices → Kannada
- macOS: System Settings → Accessibility → Spoken Content → System Voice → Manage Voices
